# План разработки: «БекОфисные приключения»

Юмористическая обучающая текстовая игра для сотрудников IT-компании (гемблинг-домен).
Игрок — руководитель виртуального продукта, отражает угрозы и держит ресурсы в норме
в течение рабочего периода (10–15 минут, ~12–18 ходов).

---

## 1. Выбранный стек и почему

| Решение | Выбор | Причина |
|---|---|---|
| Движок | **Полностью скриптовый** (контент в данных) | Бесплатный хостинг на GitHub Pages, офлайн, без ключей/сервера, детерминированный и тестируемый баланс |
| Язык/сборка | **TypeScript + Vite** | Лёгкий бандл, мгновенный dev-сервер, типобезопасность контента |
| UI | Vanilla TS (без фреймворка) | Игра — это один экран с обновляемыми блоками; React избыточен |
| Стили | CSS (один файл, CSS-переменные) | Минимум зависимостей |
| Хостинг | **GitHub Pages** через **GitHub Actions** | Авто-деплой при push в основную ветку |
| Хранение прогресса | `localStorage` | Сохранение текущей сессии без сети |
| Учёт игроков и результаты | **Supabase** (Postgres + REST + RLS) | Лидерборд и история партий без своего сервера; клиент ходит по публичному `anon`-ключу, доступ ограничен политиками RLS |
| Тесты | **Vitest** | Тот же тулинг, что и Vite; покрываем движок и баланс |

> Supabase не ломает «статичность»: фронтенд остаётся на GitHub Pages, а Supabase —
> это управляемый backend-as-a-service. `anon`-ключ публичный по дизайну (его можно
> держать в клиенте), безопасность обеспечивается **Row Level Security**, а не секретностью ключа.

> Принцип: вся «логика ведущего» (тон, события, последствия) переносится из роли LLM
> в **данные + детерминированные правила**. Текст пишется заранее, движок только
> выбирает события, считает ресурсы и применяет эффекты.

## 2. Архитектура

```
Игрок → UI (рендер хода) → выбор варианта
                ↓
          GameEngine.applyChoice()
                ↓
   ┌────────────┴─────────────┐
   │  - применить эффекты      │
   │  - начислить отложенные   │  (скрытые риски)
   │  - проверить условия конца │
   │  - выбрать следующее событие│
   └────────────┬─────────────┘
                ↓
        новое состояние → UI
```

Чистое разделение:
- **Данные** (`/src/content/*`): события, угрозы, тексты, эффекты. Не содержат логики.
- **Движок** (`/src/engine/*`): чистые функции над состоянием. Без DOM.
- **UI** (`/src/ui/*`): рендер состояния и обработка кликов. Без правил игры.

Это даёт тестируемость (движок — чистый) и лёгкость правки контента (только данные).

## 3. Модель данных

### 3.1 Ресурсы (`Resources`)
```ts
interface Resources {
  clientTrust: number;   // Доверие клиентов  (старт 100)
  peerTrust: number;     // Доверие коллег    (старт 100)
  techDebt: number;      // Технический долг  (старт 0, растёт = плохо)
  team: number;          // Команда           (старт 100)
  budget: number;        // Бюджет            (старт 100)
}
```
Все, кроме `techDebt`, лучше держать в 0..100. `techDebt` 0..100, где высокий — плохо.

### 3.2 Защитные механизмы (`Defenses`)
8 процессов, у каждого уровень 0..3. Старт — 0 (или 1 для базовых, обсуждаемо).
```ts
type DefenseKey =
  | 'codeReview' | 'riskAssessment' | 'qa' | 'postReleaseAcceptance'
  | 'monitoring' | 'backup' | 'incidentResponse' | 'changeManagement';
```
Эффект уровня: снижает урон/вероятность срабатывания связанных угроз
(`mitigation = level * COEFF`). Прокачка стоит **бюджет** и иногда **команду** (время).

### 3.3 Угрозы (`Threat`)
```ts
interface Threat {
  id: string;
  title: string;
  tier: 'common' | 'rare' | 'boss';
  intro: string;                      // описание ситуации (юмор + реализм)
  defendedBy: DefenseKey[];           // какие процессы уменьшают урон
  baseImpact: Partial<Resources>;     // урон, если не защититься
  options: ChoiceOption[];            // 3–4 варианта реакции
}
```

### 3.4 Вариант действия (`ChoiceOption`)
```ts
interface ChoiceOption {
  text: string;
  immediate: Partial<Resources>;      // мгновенные изменения ресурсов
  defenseUpgrade?: DefenseKey;        // если вариант = вложиться в процесс
  delayed?: DelayedEffect[];          // скрытые риски (см. §4)
  resolvesThreat: boolean;            // считается ли угроза «отражённой»
  feedback: string;                   // последствие (показывается после выбора)
}
```

### 3.5 Отложенный/скрытый эффект (`DelayedEffect`)
```ts
interface DelayedEffect {
  afterTurns: number;                 // через сколько ходов проявится
  text: string;                       // «костыль из прошлого вторника напомнил о себе…»
  impact: Partial<Resources>;
  triggersThreatId?: string;          // может породить новую угрозу (напр. сбой БД)
}
```

### 3.6 Состояние игры (`GameState`)
```ts
interface GameState {
  turn: number;
  resources: Resources;
  defenses: Record<DefenseKey, number>;
  pendingDelayed: { dueTurn: number; effect: DelayedEffect }[];
  threatsTotal: number;               // сколько угроз за партию
  threatsFaced: number;
  threatsDefended: number;            // для метрики «85%»
  usedThreatIds: string[];            // чтобы не повторять
  log: TurnRecord[];                  // для финального разбора
  status: 'playing' | 'won' | 'lost';
  lossReason?: string;
  rng: () => number;                  // seedable RNG (детерминизм/тесты)
}
```

## 4. Игровой цикл (движок)

Каждый ход:
1. **Проявление отложенных эффектов** — применить `pendingDelayed` с `dueTurn === turn`
   (это и есть «скрытые риски, всплывающие через несколько ходов»).
2. **Выбор события**: с вероятностью — случайное юмористическое событие (§6),
   иначе — угроза по расписанию tier'ов (§5).
3. **Рендер**: описание + блок ресурсов + 3–4 варианта.
4. **Выбор игрока** → `applyChoice`:
   - применить `immediate`;
   - применить `defenseUpgrade` (списать бюджет/команду);
   - уменьшить `baseImpact` угрозы по сумме уровней `defendedBy` (если игрок не закрыл угрозу прямым действием);
   - поставить `delayed` в очередь;
   - обновить `threatsFaced/threatsDefended`;
   - показать `feedback`.
5. **Проверка условий конца** (§7).
6. **Переход** к ходу `turn + 1`.

UI **ждёт** выбора игрока и не продолжает сам (соответствует ТЗ).

## 5. Расписание угроз и баланс

- Партия: **12–18 ходов** (подбирается под 10–15 мин).
- Структура tier'ов: преимущественно `common`, вкрапления `rare`, **1–2 boss'а**
  (один ближе к финалу, напр. «Релиз в пятницу» / «Высокий сезон»).
- Угрозы из ТЗ заполняются в `/src/content/threats/`:
  - **Обычные**: ошибка конфигурации, кривой релиз, ошибка в постановке задачи,
    переполнение очередей, отсутствие проверки совместимости, деградация
    производительности, человеческий фактор, отказ внешнего провайдера.
  - **Редкие**: потеря бэкапов, сбой БД, сетевой сбой, блокировка домена, рост нагрузки.
  - **Боссы**: Внутренний Разгильдяй, Релиз в пятницу, Высокий сезон,
    Критическая уязвимость, Массовая блокировка.
- **Связки процесс↔угроза** (для обучающих целей, §8): напр. «ошибка конфигурации»
  гасится `changeManagement` + `monitoring`; «кривой релиз» — `qa` +
  `postReleaseAcceptance`; «потеря бэкапов» — `backup`; «критическая уязвимость» —
  `riskAssessment` + `incidentResponse`.
- Баланс настраивается константами в `/src/engine/balance.ts` и проверяется
  симуляцией (§9). Цель: победа достижима «правильной» игрой, но не гарантирована.

## 6. Корпоративный юмор (случайные события)

Пул нейтральных/флейворных вставок (из ТЗ + расширим), часть с мелким эффектом:
- «У меня локально работает», «TODO: потом исправим», «фича маленькая»,
  «баг по вторникам», «срочная задача через 5 минут после начала отпуска».
Реализуются как лёгкие `Threat`-подобные карточки или как баннер перед ходом.
Тон: лёгкий корпоративный сарказм без токсичности; без магии/фэнтези/фантастики.

## 7. Условия завершения и счёт

**Поражение** (любое из):
- любой ресурс доверия (клиенты/коллеги) ≤ 0 → «потеря клиентов / уход партнёра»;
- наступление «жёсткого» отложенного события: утечка данных / штраф регулятора /
  простой > 4 ч (эти исходы = следствие проигнорированных рисков);
- `team` ≤ 0 (команда выгорела) или `budget` < 0 без возможности реакции.

**Победа**:
- пройдены все ходы периода **и** `threatsDefended / threatsFaced ≥ 0.85`
  **и** не случилось критического инцидента.

**Финальный экран** (по ТЗ):
- итоговый счёт (формула из остатков ресурсов − techDebt + бонус за % отражённых угроз);
- разбор решений (из `log`);
- какие процессы помогли больше всего (по накопленному `mitigation`);
- какие риски проигнорированы (нереализованные `defendedBy` → сработавший урон).
Выводы §8 **не сообщаются напрямую** — следуют из разбора.

---

## 8. Обучающие цели (зашиты в баланс, не в текст)

Достигаются механикой, а не нравоучениями:
1. Оценка рисков заранее → `riskAssessment` резко гасит боссов.
2. Пост-релизная приёмка → `postReleaseAcceptance` ловит «кривой релиз».
3. Тестирование дешевле аварий → стоимость прокачки `qa` << урона от дефектов.
4. SDLC не просто так → пропуск процессов копит `techDebt` и отложенные риски.
5. Скорость без контроля → «быстрые костыли» дают мгновенный плюс, но `delayed` минус.
6. Конфиг опаснее кода → у «ошибки конфигурации» урон выше, чем у «ошибки кода».

## 9. Учёт игроков и таблица результатов (Supabase)

Цель: после партии игрок указывает ник, результат сохраняется, и показывается
**глобальная таблица лидеров** (топ-N) + личная история.

### 9.1 Модель данных (Postgres)

```sql
-- Игроки (лёгкая идентификация по нику + клиентский id, без авторизации)
create table players (
  id          uuid primary key default gen_random_uuid(),
  nickname    text not null check (char_length(nickname) between 2 and 24),
  client_uid  text not null,            -- генерится на клиенте, хранится в localStorage
  created_at  timestamptz not null default now(),
  unique (client_uid)
);

-- Завершённые партии
create table runs (
  id                uuid primary key default gen_random_uuid(),
  player_id         uuid not null references players(id) on delete cascade,
  score             integer not null check (score >= 0),
  outcome           text not null check (outcome in ('won','lost')),
  loss_reason       text,
  turns             integer not null,
  threats_faced     integer not null,
  threats_defended  integer not null,
  defense_peak      jsonb not null default '{}',  -- финальные уровни процессов
  duration_seconds  integer,
  created_at        timestamptz not null default now()
);

-- Публичная вьюшка лидерборда (отдаём только нужное, без client_uid)
create view leaderboard as
  select r.id, p.nickname, r.score, r.outcome, r.turns,
         r.threats_defended, r.threats_faced, r.created_at
  from runs r join players p on p.id = r.player_id
  where r.outcome = 'won'
  order by r.score desc, r.created_at asc;
```

### 9.2 Доступ и безопасность (RLS)

- Включить **Row Level Security** на `players` и `runs`.
- Политики для роли `anon`:
  - `players`: `insert` разрешён (с проверкой длины ника); `select` — только свои строки по `client_uid` (или закрыт, читаем имена через вьюшку).
  - `runs`: `insert` разрешён; прямой `select` закрыт — чтение только через `leaderboard`.
  - `leaderboard` (view): публичное чтение (`grant select`).
- `anon`-ключ кладётся в клиент (он публичный). **Сервисный ключ в репозиторий не попадает никогда.**

### 9.3 Защита от накрутки (важная оговорка)

Чисто клиентский счёт **можно подделать** — это неизбежно для статики. Подход:
- Базово: трактуем лидерборд как «казуальный», добавляем серверные `check`-ограничения
  (диапазоны `score`/`turns`, `threats_defended ≤ threats_faced`).
- При желании усилить — **Supabase Edge Function** (Deno): принимает компактный лог
  ходов, **переигрывает партию тем же детерминированным движком** на сервере и сам
  считает счёт; клиенту запрещается прямой `insert` в `runs`. Это выносится в опц. этап
  (движок уже чистый и переиспользуемый — см. §2).

### 9.4 Конфигурация ключей

- `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` — через `.env` локально и
  **GitHub Actions Secrets** при сборке (инъекция на этапе `build`).
- `.env` в `.gitignore`; в репозитории — `.env.example` с пустыми значениями.
- Клиент `@supabase/supabase-js` инициализируется в `/src/services/supabase.ts`.

### 9.5 Поведение офлайн / без сети

Игра полностью играбельна без Supabase: если ключи не заданы или запрос упал —
лидерборд деградирует до локального (`localStorage`), игра не блокируется.

## 11. UI/UX

Один экран, верстаемый блоками (полностью соответствует «Формату ответа» из ТЗ):
- Шапка: «Ход N» + прогресс периода.
- 📊 Панель ресурсов: Доверие клиентов / Доверие коллег / Технический долг /
  Команда / Бюджет (с цветовой индикацией и анимацией изменения).
- Карточка ситуации: описание.
- Кнопки вариантов (3–4).
- После выбора: панель последствий → кнопка «Дальше».
- Финальный экран с разбором + **ввод ника и таблица лидеров** (топ-N + личная история).
Адаптив под мобильные, тёмная тема, минимум зависимостей.

## 12. Деплой (GitHub Pages + Supabase)

`.github/workflows/deploy.yml`: на push в основную ветку — `npm ci && npm run build`,
публикация `dist/` через `actions/deploy-pages`. В `vite.config.ts` задаётся
`base: '/itgame/'`. В настройках репозитория Pages → Source: GitHub Actions.

**Supabase:**
- Схема (`supabase/schema.sql`) применяется в проекте Supabase один раз (SQL Editor или CLI).
- `VITE_SUPABASE_URL` и `VITE_SUPABASE_ANON_KEY` хранятся в **GitHub Actions Secrets**
  и пробрасываются в шаг `build` как env. Локально — через `.env` (в `.gitignore`).
- Сервисный/`service_role` ключ **не используется на клиенте и не коммитится**.

> Аккаунт Supabase предоставит владелец репозитория. Нужны: `Project URL`, `anon public`
> ключ и доступ к SQL Editor для применения схемы. Эти данные дать **не в чат и не в коммит**,
> а сложить в GitHub Secrets (`Settings → Secrets and variables → Actions`).

## 13. Дорожная карта

- **M0 — Каркас**: Vite+TS, структура папок, пустой экран, CI-деплой на Pages.
- **M1 — Движок**: типы, `applyChoice`, отложенные эффекты, условия конца, RNG + тесты.
- **M2 — Контент v1**: все обычные угрозы + 2 босса + случайные события + защиты.
- **M3 — UI**: рендер хода, панель ресурсов, финальный разбор, стили/адаптив.
- **M4 — Баланс**: симуляция партий (Vitest), подбор констант под 85%/10–15 мин.
- **M5 — Лидерборд (Supabase)**: схема + RLS, клиент `services/`, экран ввода ника
  и таблицы результатов, фоллбэк на localStorage без сети.
- **M6 — Полировка**: рекорды, звук/анимации (опц.), README; опц. Edge Function для
  серверной валидации счёта.

## 14. Открытые вопросы (на потом)

- Стартовые уровни защит: все 0 или базовые на 1?
- Нужна ли «новая угроза, порождённая отложенным эффектом» (каскады)?
- Локализация: только RU или RU/EN?
- Сложности (easy/normal/hard) через `balance.ts`?
- Лидерборд: глобальный для всех или по «компаниям»/командам?
- Нужна ли серверная валидация счёта (Edge Function) сразу, или достаточно RLS + `check`?
- Уникальность ника: разрешить дубликаты или привязывать к `client_uid`?
